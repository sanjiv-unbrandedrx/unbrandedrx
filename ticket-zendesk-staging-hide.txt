TICKET: Hide Zendesk Support Widget on /staging/ URLs
Assigned to: Marian / Sanjiv
Priority: Medium
Date: 2026-03-13

CONTEXT:
The Zendesk support button (chat widget) loads globally on every page via
Google Tag Manager. On staging pages (/staging/*), particularly the
questionnaire at /staging/visit, the floating support button overlaps with
the navigation controls (Back/Continue buttons) and treatment plan UI on
mobile devices, creating a poor user experience.

We've added a client-side workaround that hides the widget via the Zendesk
JS API on the visit page, but this is fragile — it races with the widget
load and may flash briefly. The proper fix is to exclude /staging/ URLs
from loading the Zendesk snippet entirely.

REQUESTED CHANGE:
In Google Tag Manager (or Zendesk admin), configure the Zendesk widget to
NOT load on any URL containing "/staging/" in the path.

OPTIONS:
1. GTM Trigger Exception — Add a "Page Path does not contain /staging/"
   condition to the Zendesk tag trigger in GTM. This is the easiest fix.
2. Zendesk Admin — Use Zendesk's built-in page targeting to exclude
   URLs matching */staging/*.
3. Either approach works. GTM is preferred since it's already where the
   script is managed.

AFTER THIS IS DONE:
We can remove the client-side Zendesk hide workaround from
src/components/staging/visit/questionnaire-shell.tsx (the useEffect that
calls window.zE("messenger", "hide")).
